# MLOps에 대한 소개

## 1. 개요 (Overview)
인공지능(AI) 기술이 실험실을 넘어 실제 산업 현장에 적용됨에 따라, 머신러닝 모델의 **일관된 성능 유지** 와 **운영 효율화** 가 기업의 핵심 역량으로 부상했습니다. 실험실에서 검증된 모델이 실제 서비스로 이어지지 못하거나, 배포 후 시간이 지나며 성능이 급격히 저하되는 문제가 반복되면서, 모델의 전체 생애주기를 체계적으로 다루는 **MLOps**가 주목받게 되었습니다.

---

## 2. MLOps란?

### 2.1 MLOps의 정의와 워크플로우
MLOps는 단순히 모델을 개발하는 것을 넘어, 모델이 **실제 서비스 환경** 에서 안정적으로 동작하도록 만드는 것이 핵심입니다. 

#### 2.1.1 ML 생애주기 단계

> **흐름:** ① 문제 정의 → ② 데이터 수집 → ③ 데이터 준비 → ④ 모델 학습 → ⑤ 모델 평가 → ⑥ 모델 배포 → ⑦ 모니터링 → ⑧ 재학습 ↩ (다시 ②로 순환)

| 단계 | 주요 활동 내용 | 핵심 목표 |
| :--- | :--- | :--- |
| **① 문제 정의** | 비즈니스 목표를 ML 문제로 정의, 성공 지표 설정, 데이터·기술 실현가능성 검토 | 비즈니스 정합성 및 실현가능성 확보 |
| **② 데이터 수집** | 원천 데이터 수집, 데이터셋 버전 관리, 데이터 카탈로그 등록 및 출처 기록 | 데이터 출처(Provenance) 및 계보(Lineage) 확보 |
| **③ 데이터 준비** | 전처리, 피처 엔지니어링, Feature Store(예: Feast) 적재 | 학습/서빙 데이터 일관성 |
| **④ 모델 학습** | 실험 관리, 하이퍼파라미터 튜닝, 소스 제어 | 재현성(Reproducibility) 확보 |
| **⑤ 모델 평가** | 오프라인 성능 검증, A/B 테스트 설계 | 품질 게이트(Quality Gate) 통과 |
| **⑥ 모델 배포(Serving)** | 모델 레지스트리 등록, 추론 서버 구성(예: BentoML), 배포 롤아웃(카나리/롤백) | 안정적 서비스 노출 |
| **⑦ 모니터링** | 성능 추적, 드리프트 감지, 알림 | 실시간 서비스 안정성 보장 |
| **⑧ 재학습(CT)** | 자동 재학습 트리거, 모델 폐기 정책 | 모델 노후화(Decay) 대응 |

#### 2.1.2 공통 관리 영역 (Cross-cutting Concerns)

생애주기의 특정 단계가 아니라 **전 단계에 걸쳐 적용**되는 영역입니다.

| 영역 | 주요 활동 내용 | 핵심 목표 |
| :--- | :--- | :--- |
| **CI/CD** | 코드·데이터·모델의 통합 및 배포 자동화, 단계별 테스트 | 신속한 모델 업데이트(Agility) |
| **실험 관리** | 학습 코드·파라미터·메트릭 추적 및 비교 | 재현성 및 비교 가능성 |
| **거버넌스** | 법적 규제 준수, 윤리적 편향 관리, 감사 로그 | 모델의 신뢰성 및 투명성 증명 |
| **보안/접근 제어** | 데이터·모델 접근 권한 관리, 시크릿 관리 | 컴플라이언스 및 정보 보호 |

### 2.2 DevOps와 MLOps의 차별점
MLOps는 기존 소프트웨어 공학의 DevOps 개념을 계승하지만, **'데이터'**라는 가변적 요소로 인해 더 높은 복잡성을 가집니다.

* **코드 vs 데이터:** DevOps는 코드 버전 관리가 핵심이나, MLOps는 **코드 + 데이터 + 모델** 세 가지 요소의 버전을 동시에 관리해야 합니다.
* **성능 감쇠:** 일반 소프트웨어는 버그가 없는 한 성능이 유지되나, ML 모델은 시간이 흐름에 따라 데이터 환경이 변하며 성능이 저하되는 **'모델 노후화(Decay)'**가 발생하므로 **지속적 학습(CT, Continuous Training)** 이 필수적입니다.

## 3. 시장 및 정치적 맥락 (Market & Political Context)

### 3.1 지정학적 리스크 및 기술 주권 (Sovereign AI)
* **전쟁 및 공급망 위기:** 글로벌 분쟁으로 인한 GPU 및 서버 자원의 수급 불안정은 모델 학습 비용의 급등을 초래합니다. MLOps는 분산 학습 및 자원 스케줄링을 통해 **한정된 컴퓨팅 자원을 최적화**하는 전략적 수단이 됩니다.
* **기술 블록화:** 국가 간 기술 패권 경쟁이 심화되면서 특정 국가의 클라우드 플랫폼에 대한 의존도를 낮추려는 움직임이 강해지고 있습니다. 오픈소스 기반의 MLOps 체계는 기술적 자립도를 높이는 데 기여합니다.

### 3.2 자국 우선주의 기조와 AI 규제
* **자국 우선주의 정책:** 주요국의 자국 기술 보호 기조 강화 시, 데이터 유출 방지와 로컬 인프라 기반의 AI 운영이 중요해집니다. 이는 클라우드 네이티브를 넘어선 **하이브리드 MLOps** 수요를 창출합니다.
* **글로벌 규제(EU AI Act 등):** AI의 결정 과정에 대한 '설명 책임'이 법제화되고 있습니다. MLOps 거버넌스 항목은 모델의 모든 생성 과정을 기록하여 법적 과징금 리스크를 선제적으로 방어하는 도구가 됩니다.

## 4. 리스크 및 대응 (Risk & Mitigation)

| 리스크 유형 | 상세 리스크 | 대응 전략 |
| :--- | :--- | :--- |
| **기술적 리스크** | 데이터 드리프트에 따른 모델 성능 급락 | **자동 재학습 파이프라인(CT)** 및 성능 임계치 기반 알림 구축 |
| **운영적 리스크** | 데이터 과학자와 엔지니어 간의 소통 단절 | 표준화된 **Feature Store** 및 협업 도구 도입을 통한 사일로 제거 |
| **비즈니스 리스크** | 불투명한 인프라 비용 및 ROI 저하 | **FinOps**를 결합한 리소스 모니터링으로 컴퓨팅 비용 가시화 |

## 5. 결론 및 제언 (Conclusion)

MLOps는 선택이 아닌 **AI 비즈니스의 생존을 위한 필수 인프라**입니다.

1.  **전략적 자산으로서의 접근:** MLOps를 단순한 운영 도구가 아닌, 데이터 기반의 의사결정을 자동화하고 가속화하는 기업의 전략적 자산으로 인식해야 합니다.
2.  **거시 환경에 유연한 설계:** 특정 벤더에 종속되지 않는 유연한 아키텍처를 채택하여 지정학적 리스크와 정치적 규제 변화에 즉각 대응할 수 있는 체력을 갖춰야 합니다.
3.  **점진적 고도화:** 모든 과정을 한 번에 자동화하기보다, 수동 운영(Level 0)에서 시작하여 파이프라인 자동화(Level 1), 완전 자동화(Level 2)로 단계를 밟아 나갈 것을 제언합니다. (참고: Google Cloud, *MLOps: Continuous delivery and automation pipelines in machine learning*)

## 부록: 참고 용어

**문제 정의 & 데이터**

- 정합성(Alignment): 비즈니스 목표와 ML 문제 정의가 어긋나지 않게 맞물려 있는 상태 (예: "이탈률 감소"라는 KPI가 "로그인 예측"이 아닌 "결제 미발생 확률 예측"으로 번역됨)
- 실현가능성(Feasibility): 보유한 데이터·기술·예산으로 그 ML 문제를 실제로 풀 수 있는지에 대한 판단
- 출처(Provenance): 데이터가 어떤 원천 시스템에서, 언제, 어떻게 수집되었는지를 추적할 수 있는 상태 — "어디서 시작됐나"
- 계보(Lineage): 데이터의 원천부터 가공·사용처까지 전체 흐름을 양방향으로 추적할 수 있는 상태 — "어떻게 흘렀나"
- Feature Store: 모델 입력 피처를 중앙에서 정의·저장·공급해 학습과 서빙이 동일한 값을 가져오게 만드는 ML 전용 데이터 플랫폼

**학습 & 평가**

- 재현성(Reproducibility): 동일한 코드·데이터·환경·파라미터로 학습했을 때 동일한 모델을 만들어낼 수 있는 성질
- 실험 관리: 모델 학습 시도 한 번 한 번을 누가·언제·무엇으로 했고 어떤 결과가 나왔는지 자동으로 기록·비교·재현하게 만드는 활동
- 오프라인 성능 검증: 실제 사용자가 영향을 받기 전, 과거 정답 데이터를 이용해 통제된 환경에서 수행하는 평가
- A/B 테스트: 실서비스 트래픽 일부를 신모델에 분배해 실제 비즈니스 지표 차이를 측정하는 온라인 검증 방식
- 품질 게이트(Quality Gate): 정의된 지표 임계치를 통과한 모델만 다음 단계로 넘기는 자동 합격/불합격 판정 기준

**배포**

- 모델 레지스트리 등록: 학습 끝난 수많은 시도 중에서 '운영에 쓸 자격이 있는 모델'만 골라 이름·버전·메타데이터·승인 이력과 함께 공식 등재하는 행위
- 추론 서버: 등록된 모델을 로드해 HTTP/gRPC API로 노출하고 입력 검증·배칭·메트릭 노출을 담당하는 서비스 (예: BentoML, MLflow Models, Seldon Core)
- 카나리/롤백: 신모델을 일부 트래픽(예: 5%)에 우선 노출해 검증 후 확대(카나리)하거나, 문제 발생 시 이전 버전으로 즉시 복귀(롤백)하는 안전 배포 절차

**모니터링 & 재학습**

- 모델 노후화(Decay): 시간이 지나며 데이터 환경이 변해 학습 시점의 모델 성능이 점차 떨어지는 현상
- 드리프트(Drift): 모델이 학습했던 시점의 규칙이 시간이 지남에 따라 변하여 기존 모델의 성능이 급격히 저하되는 현상
- 컨셉 드리프트(Concept Drift): 입력과 정답 사이의 상관관계(P(Y|X))가 변해 예측 로직 자체가 유효성을 상실하는 현상 — 소비자 트렌드·시장 규칙 등 '정답을 도출하는 기준' 자체가 바뀔 때 발생
- 데이터 드리프트(Data Drift): 예측 로직은 그대로지만 입력 데이터의 통계적 분포(P(X))가 학습 시점과 달라지는 현상 — 신규 사용자 유입이나 수집 환경 변화 등 '입력값의 성격'이 변할 때 발생
- 지속적 학습(CT, Continuous Training): 새로운 데이터로 모델을 자동 재학습해 노후화에 대응하는 자동화 파이프라인
